Recent Interesting Projects:
GWSMaterialUIKit

As of Fall 2016, Gradle builds of android projects are still slow when you have a lot of aars to generate or load. This occurs in projects targeting API 16 to 21 as we still have a lot of back-ported UI features to handle.


Thus, to meet that use case and solve the diamond dependency problem went searching for the best solution in back-porting to adjust for theme new featues and ripple. The best solution was the Material project which only needed some light code clean-up to be production ready.


Combined that with some code of my own, custom fonts and custom font loading, and some of my own Material resoruces to make a complete Material UI library for targeting api levels before Lollipop.


Not very sexy, but often solutions are not sexy just in fact mundane and boring but needed. But, because this already tackled I can concentrate on other important things like the architecture of the application. AND, when creating a custom View or UI widget I just extend from this library and get ripple, tint, themes automatically without having to think about it.


Benefits:

-properly integrate with Android Support Library internals to support theme new features with api 21

-one compact aar instead of many small aars to imporve gradle build speeds

-includes my material resource extras; keyline spacing, full monochromatic material color pallete, etc.

-includes custom fonts and the proper memory leak free way fo loading them via both view decorator and a typefacehelper to be able to load them via special xml tags or programmtically

-private resources back-ported to avoid support library not compatible changes

GWSImmutableVMKit

With Google Data Binding it give us an easy way to two-way bind UI events to a view-model. View-Models are extremely flexible in that you can fast prototype with almost no application architecture at all or even add architecture components as the application development matures.


But with this new flexibility would it not be nice to have somewhat lite application responsiveness without the heavy RxJava learning curve? Turns out if you recreate collections to use unodified Maps and Lists than you can in fact create an immutable View-Model for storage using Android's bundle-parceable machinery and thus have an automatic way that device orientation changes are handled.


The original discover of the immutable view-state gem did not realize that one could extend it to Google Data Binding. Thus, I cleaning up the code and extending it to Google Data Binding usage.


Benefits:

-Lite App Responsiveness via immutable of the UI events and background events flow to immutable view-state

-Automatic Device orientation handling so the developer does not have to think that hard

-Flexible enough that I can extend to any application architecture pattern

-Turns Activities and Fragments into View Holder patterns

GWSTheadNetKit

The Apache Client Api was deprecated in api 21. And one does not want to have to depend upon interfacing with different Volley network versions for a networking library. The solution is to use OkHttp3 libraries and set up a library that takes the pain out of the networking as far as thread pools are concerned. That way one can set an initialization method call in the extended application class without the application class being cluttered by boilerplate code. As is often the case found basic code that needed cleaned up to use, mainly proper log wrapping to avoid leaking any information in production applications.


Benefits:

-one version of the library to interface with all android OS versions

-integrated with thread pools and executors for proper threading

-OkHttp uses okio to fine tune memory GC management

-one less headache

GWSThreadNetKit Repo

Benefits:
